3. Database Schema Mapping

Every entity maps locally and remotely. Local uses IndexedDB v3. Remote uses Supabase Postgres.

4. Data Sync Flow (OCC & Tombstones)

The SyncService runs entirely in the background. It employs Optimistic Concurrency Control (OCC) using monotonically increasing integers (version) to protect against split-brain scenarios and data overwrites.

flowchart TD Start((Local Change)) --> LocalInc["Increment local version
(v = v + 1)"] LocalInc --> Push[Push Payload to Supabase] Push --> ServerCheck{Server Version vs Payload Version?} ServerCheck -->|Server < Payload| Accept[Server Accepts Upsert] ServerCheck -->|Server > Payload| Conflict[Conflict detected!] Conflict --> Pull[Pull Remote Record] Pull --> MergeCheck{Which is newer?} MergeCheck -->|Remote updated_at > Local| Adopt[Adopt Remote Record entirely] MergeCheck -->|Local updated_at > Remote| Overwrite[Overwrite Server Record] Adopt --> UpdateLocal[Update IndexedDB]

Tombstone Architecture: When a user deletes a note locally, it is erased from IndexedDB. Simultaneously, `core/db.js` fires `SyncService.deleteNoteFromCloud()`. Instead of hard-deleting in the cloud, an entry is inserted into the `tombstones` table. When Device B connects, it fetches tombstones first and scrubs its local DB.

5. Zero-Knowledge Crypto Flow (Password Vault)

The heart of the password manager security model leveraging the Web Crypto API via password-manager/crypto.js. Supabase only ever sees meaningless ciphertext.

flowchart LR MP[Master Password
User's Brain] -->|PBKDF2-SHA256
600,000 Iterations| KEK(Key Encryption Key
Ephemeral RAM) VaultRaw[Raw JSON Vault] -->|AES-GCM 256| VaultEnc[Encrypted Vault Blob
Ciphertext] DEK(Data Encryption Key
AES-GCM) -->|Encrypts| VaultEnc KEK -->|Encrypts| DEK DEK --> EDEK[Encrypted DEK] VaultEnc --> DB[(Supabase DB)] EDEK --> DB

Cryptographic Steps in Detail:

  • Creation: When a user creates a Vault, a 256-bit symmetric Data Encryption Key (DEK) is generated. The empty vault JSON is encrypted with this DEK.
  • Derivation: The user's Master Password is hashed using PBKDF2 (600,000 rounds) + a 16-byte random salt to generate the Key Encryption Key (KEK).
  • Wrapping: The KEK is used exclusively to encrypt the DEK. The output is the Encrypted DEK (EDEK). The EDEK and Vault Ciphertext are saved to the cloud.
  • Decryption (Device B): Device B downloads the EDEK. User types Master Password -> KEK is derived -> EDEK is decrypted yielding the DEK -> Vault is decrypted.
